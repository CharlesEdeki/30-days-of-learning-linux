# Day 17 - Filesystem Capacity, Repair, and Special Purpose Filesystems

## Objective
Learn how to measure filesystem usage, check and repair broken
filesystems safely, and understand why some filesystems have no
physical disk behind them at all.

---

## What I Learned

### Filesystem Capacity
- `df` shows every mounted filesystem, its size, used space,
  available space, usage percentage, and mount point
- `df -h` makes output human readable, showing GB and MB instead
  of raw block counts
- `df .` limits output to the filesystem holding the current
  directory
- Used plus Available never equals total capacity. The missing
  space is reserved blocks held back for the superuser so system
  services keep running even when regular users hit the disk limit
- `du -s *` shows disk usage of everything in the current directory
- `du -sh *` does the same with human readable sizes
- Running `du -s` in the day-17 directory returned 920KB, a small
  directory with a few files

### Filesystem Repair
- Filesystems are complex database structures. The kernel needs
  data on disk to be consistent for them to work correctly
- Unclean shutdowns, power cuts, forced restarts, or pulling a
  drive mid-write can leave the filesystem cache out of sync
  with what is physically on disk
- `fsck` is the tool for checking and repairing filesystems
- CRITICAL: never run fsck on a mounted filesystem. The kernel
  is actively reading and writing it. Running fsck at the same
  time can corrupt data worse than the original problem
- fsck runs five passes: inodes and blocks, directory structure,
  directory connectivity, reference counts, group summary
- Files with unrecoverable names are placed in lost+found with
  a number as the filename. The original name is gone
- `fsck -p` fixes ordinary problems automatically without asking.
  Linux runs this variant at boot time
- `fsck -n` checks without making any changes. Safe for assessing
  damage before committing to a repair
- ext3 and ext4 journals record what the filesystem was about to
  do before doing it, enabling clean recovery after unclean
  shutdowns without manual fsck in most cases
- `e2fsck -fy /dev/device` flushes the journal on ext3 and ext4
- If only the superblock is corrupt, backup superblocks exist.
  `mkfs -n` lists their locations. `fsck -b num` replaces the
  damaged one with a backup

### Special Purpose Filesystems
- Not all filesystems represent physical storage. The filesystem
  interface is universal enough to represent process data,
  hardware attributes, memory, and remote storage
- proc: mounted on /proc. Each numbered directory is a running
  process. Content generated by kernel in memory, nothing on disk.
  /proc/cpuinfo used on Day 9 is one example
- sysfs: mounted on /sys. Exposes hardware device information.
  Covered in Day 12. Also entirely in memory
- tmpfs: mounted on /run and other locations. Uses physical memory
  and swap as backing store. Data lives in RAM not disk. The three
  tmpfs entries visible in df -h output are exactly this
- squashfs: read-only compressed filesystem extracted on demand.
  Used by snap package management under /snap
- overlay: merges directories into a composite view. Docker and
  containers use this to layer filesystems without copying entire
  base images
- FUSE (File System in User Space) lets regular user space programs
  present a filesystem interface without touching the kernel.
  Powers sshfs (remote server as local directory) and s3fs
  (S3 bucket as local directory)

## Key Takeaways
- Never run fsck on a mounted filesystem. This is not a
  suggestion, it is the rule
- Reserved blocks mean root can always write even when the
  disk is full for regular users. System services stay alive
- The df -h output from the VM showed three tmpfs filesystems
  already running before today's reading made them visible.
  The filesystem interface unifies disk storage, RAM, hardware
  attributes, and remote storage under the same directory tree
- ext3 and ext4 journals make corruption far less common but
  proper shutdown procedure still matters
- Backups are not optional. The worst case scenarios in fsck
  documentation exist because people encountered them without
  backups
- FUSE connects directly to data engineering work. s3fs and
  sshfs mean cloud storage and remote servers can be accessed
  with the same commands used on local files

## Commands Used

```bash
df -h                          # human readable filesystem usage
df .                           # filesystem for current directory
du -s                          # disk usage of current directory in KB
du -sh *                       # human readable usage per item
fsck /dev/device               # check and repair filesystem
fsck -p /dev/device            # automatic repair, no prompts
fsck -n /dev/device            # check only, no modifications
fsck -b num /dev/device        # restore superblock from backup num
e2fsck -fy /dev/device         # flush journal on ext3 and ext4
mkfs -n /dev/device            # list backup superblock locations
```
